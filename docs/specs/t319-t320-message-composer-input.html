T319/T320 Message Composer Input

Self-contained Clawline product and technical specification for modified Enter behavior, Catalyst message-bubble context menus, and structured composer message references. Prepared 2026-05-17 from T319 and T320 tracker text supplied by Flynn.

Decision: modified Enter is composer text editing, plain Enter remains prompt submit, and message references are visible composer tokens backed by structured message identity rather than pasted quoted text.

T319 and T320 are specified together because both features affect the prompt composer submission boundary. The implementation must preserve the distinction between visible prompt text, local composer UI tokens, and structured reference context sent with the prompt.

Goals

  • Make prompt composer keyboard behavior match standard multiline input semantics: Shift+Enter and Ctrl+Enter insert a newline into the prompt composer, while plain Enter/Return submits the prompt.
  • Add a Catalyst chat-bubble/title-area context menu opened by mouse or trackpad right-click and by tapping the title-bar area of a chat bubble.
  • Expose Insert into prompt only for user bubbles/messages; it inserts the user message contents into the prompt composer as editable prompt text.
  • Expose Reference message / quote behavior that adds a visible composer reference token, lets the user type additional prompt text, and sends the typed prompt plus structured referenced-message identity.
  • Ensure reference/quote behavior does not paste the referenced message body into visible prompt text.
  • Define enough cross-platform behavior for Web, Catalyst, iOS, and iPadOS implementation checks so platform-specific text controls do not diverge.

Non-Goals

  • No implementation, source-code edit, deployment, merge, rebase, or cleanup work is part of this spec.
  • No change to plain Enter/Return prompt submission.
  • No new message sanitization, security filtering, permissions model, or legacy compatibility layer beyond the behavior explicitly specified here.
  • No hidden prompt-text quoting by copying the referenced message body into the user's prompt draft.
  • No requirement for context menus on every platform. The product scope named by Flynn is Catalyst; other platforms receive compatibility notes and proof expectations where relevant.
  • No transcript mutation when a message is merely referenced. Referencing a message affects the composer draft and the outgoing send context only.

Terminology

T319 Keyboard Contract

The prompt composer has one submission key and two specified newline gestures. This contract applies only while focus is inside the prompt composer text editor. Other focused controls keep their existing behavior unless a separate Clawline spec assigns them composer-equivalent text editing behavior.

Input Required Behavior Must Not
Enter/Return Submit the current prompt using the existing send eligibility and failure paths. Insert a newline as its primary behavior.
Shift+Enter Insert one newline into the prompt draft at the current insertion point or selected range. Submit, blur the composer, create a transcript message, or create a placeholder send.
Ctrl+Enter Insert one newline into the prompt draft at the current insertion point or selected range. Submit, blur the composer, create a transcript message, or create a placeholder send.

When a mention picker, command palette, native context menu, or other explicitly modal surface owns Enter under another Clawline spec, that surface keeps its own documented precedence while it is open. Outside such an owning surface, the prompt composer behavior above is authoritative.

T320 Context Menu Contract

Open Gestures

  • On Catalyst, mouse right-click on a chat bubble or bubble title area opens the context menu for that bubble's message.
  • On Catalyst, trackpad secondary click on a chat bubble or bubble title area opens the same menu.
  • On Catalyst, tap in the title-bar area of a chat bubble opens the same menu.
  • The menu target is the message associated with the bubble/title area that received the gesture. The compositor/presentation layer may route hit regions against resolved bubble rectangles, but it must not reconstruct message layout topology from higher-level intent.

Menu Items

Item Visibility Action
Insert into prompt User bubbles/messages only. Insert the targeted user message contents into the prompt composer as editable text. The action does not submit.
Reference message Transcript messages that have stable identity and can be represented in reference context. Add a visible composer reference token backed by structured identity for the targeted message. The action does not paste message body text into the prompt and does not submit.

The ticket names reference/quote behavior. Product behavior is specified as Reference message unless implementation already uses a shorter visible label such as Quote; the required behavior is the token and structured identity contract, not the exact English label.

Composer Reference UX

  • After Reference message, the composer shows a visible token before or near the prompt text editing area. The token must be visually distinct from typed prompt text.
  • The token should identify the target enough for the user to understand what will be referenced, such as role/author and a short message locator or compact preview. The preview must not become submitted prompt text.
  • The user can continue typing prompt text after adding the token.
  • The user can submit a prompt with one or more reference tokens if the implementation supports multiple tokens. If implementation chooses single-reference behavior for this pass, adding a second reference must replace the first or be explicitly blocked in UI; silent duplicate or hidden references are not allowed.
  • The user can remove a reference token before submit. Removed tokens must not remain in the outgoing structured context.
  • Composer focus should move to or remain in the prompt text editor after either Insert into prompt or Reference message, so the user can keep typing.
  • Sending with only a reference token and no prompt text is allowed only if the existing composer send eligibility permits non-text context sends. If current send eligibility requires prompt text, this spec does not add a new empty-text send rule.

Data and Context Contract

The core distinction is that reference tokens are UI and reference context is structured data. The referenced message text must not be copied into the visible prompt body as the mechanism for giving the LLM context.

{
  "promptText": "User typed text from the composer",
  "references": [
    {
      "kind": "message",
      "sessionKey": "agent:main:clawline:flynn:s_...",
      "streamId": "stable stream/session identifier when distinct from sessionKey",
      "messageId": "stable transcript message identifier",
      "messageRole": "user | assistant | tool | system | terminal | status",
      "createdAt": "message creation timestamp if available",
      "clientMessageId": "local/offline identity if distinct from messageId"
    }
  ]
}
  • promptText contains only the actual composer text the user typed or inserted through text-editing actions such as Insert into prompt.
  • references contains one entry per visible reference token present at submit time.
  • sessionKey is required. Clawline session keys must remain structurally distinct and must not be collapsed to agent:main:main.
  • messageId is required when the transcript has a stable message identifier. If the local model differentiates provider message ID from local client message ID, preserve both rather than overwriting one with the other.
  • messageRole is required for downstream context assembly and menu visibility checks.
  • The context assembler may resolve the referenced message body downstream using this identity, but the composer must not implement reference behavior by pasting hidden or visible quoted text into promptText.
  • If the referenced message no longer exists at submit time, use the existing send-error pathway or block submission with an inline composer error. Do not silently send a prompt whose visible token claims a reference that is absent from structured context.

Platform Notes

Platform Required Notes

iPadOS Must validate hardware keyboard behavior for plain Enter, Shift+Enter, and Ctrl+Enter in the prompt composer. Pointer/trackpad secondary click may share Catalyst code paths if available, but this spec does not require new iPadOS context-menu gestures beyond avoiding divergence where code is shared.

UX States and Edge Cases

State Expected Behavior
Empty composer, modified Enter Insert a newline according to text-editor behavior. Existing placeholder/height behavior may update, but no send occurs.
Text selected in composer, modified Enter Replace the selected text with one newline. Do not append at the end unless normal platform selection behavior would do so.
Composer disconnected or not provisioned Modified Enter remains local text editing. Plain Enter continues to use the existing disabled/send-error gating and must not create optimistic placeholders before readiness.
Menu opened for assistant/tool/status message Do not show Insert into prompt. Show reference action only if the message has stable identity usable for reference context.
Menu opened for user message Show Insert into prompt and the reference action when stable identity exists.
Reference token added, user edits prompt Token remains attached to the draft until removed or submitted. Text edits do not mutate token identity.
Reference token removed Token disappears from UI and its identity is removed from pending send context.
Draft submitted with reference token Outgoing send contains prompt text plus structured reference context. Transcript display may show the sent prompt according to existing message rendering rules, but must not pretend quoted text was typed by the user.
Referenced message deleted or unavailable before submit Submission must not silently drop the reference. Either block with a visible composer error or use the existing send-error pathway.
Context menu opens while composer has unsent draft Existing draft is preserved. Insert into prompt inserts into the current draft using normal insertion/selection behavior; reference action adds a token without replacing text.

Architecture Boundaries

  • The composer owns draft text, reference-token UI state, and the pending send context assembled from those tokens.
  • The transcript/message bubble surface owns menu targeting and message identity lookup for the targeted bubble.
  • The send pipeline receives resolved prompt text and structured references. It must not infer references by parsing visible prompt text.
  • The presentation/compositor layer may validate bounds, transform coordinate spaces, host surfaces, and route hit regions against provided bubble/title rectangles. It must not reconstruct pane layout or message topology from layout intent.
  • The implementation must preserve the existing Clawline send readiness invariant: no optimistic local placeholder should be created before transport and active-session provisioning gates allow send.

Implementation Handoff

  • Implement exactly the specified T319/T320 behavior. Do not add additional validation, hidden prompt rewriting, new routing fallback, or broad context-menu actions unless Flynn updates this spec.
  • Use the existing composer send path for plain Enter and final prompt submit. The spec changes the key handling for modified Enter, not the send architecture.
  • Use stable transcript/session identity for references. If the current model lacks a stable identity for a message class, hide or disable reference action for that class rather than inventing unstable identity.
  • Keep Insert into prompt eligibility role-based: user messages only.
  • Keep reference/quote semantically separate from insertion: insert copies user message text into the prompt; reference creates a token and structured context.

Proof Gates

  1. Artifact gate: this HTML spec renders as a standalone document and includes goals, non-goals, requirement matrix, platform notes, data/context contract, UX states, edge cases, and implementation handoff.
  2. Keyboard gate: automated or manual tests prove plain Enter submits, Shift+Enter inserts newline, and Ctrl+Enter inserts newline in the prompt composer on relevant Web, Catalyst, iOS, and iPadOS paths.
  3. No-submit gate: pressing Shift+Enter or Ctrl+Enter creates no transcript user message, no optimistic placeholder, and no outbound send.
  4. Catalyst menu gate: mouse right-click, trackpad secondary click, and title-area tap open the message context menu for the targeted bubble.
  5. Eligibility gate: Insert into prompt appears for user messages and does not appear for non-user messages.
  6. Insertion gate: selecting Insert into prompt inserts user message contents into the composer as editable text and leaves send under user control.
  7. Reference UI gate: selecting reference/quote creates a visible composer token without pasting referenced message body into prompt text.
  8. Reference payload gate: submitting with a token sends structured reference context including session identity and message identity alongside prompt text.
  9. Removal gate: removing a token before submit removes its identity from outgoing context.
  10. Session invariant gate: active and referenced Clawline session keys remain structurally distinct and are not canonicalized into agent:main:main.

Open Questions

  • Should the visible menu label be Reference message, Quote message, or another product string? This spec treats the behavior as authoritative and the exact label as product-copy follow-up.
  • Should multiple simultaneous reference tokens be supported in the first implementation pass? If not, the implementation must make single-reference behavior explicit by replacing or visibly blocking additional references.
  • What exact downstream wire/schema field names should carry reference context if an existing Clawline envelope already has a preferred extension point? The semantic contract in this spec is required regardless of field names.
