T306 Live Agent Progress Recon

Product and architecture reconnaissance for native Clawline live current-turn progress. Prepared 2026-05-14.

Decision: native Clawline should align to upstream protocol v4 chat/progress semantics for current-prompt activity UI, not Task Ledger, as the primary source for "what the agent is doing now."

The first implementation should add an additive provider-to-native live progress event stream for Clawline sessions, then render a compact current-turn progress surface in the native clients. Existing session status/control APIs should remain snapshots for busy state, model/mode, queue depth, and control actions. Full upstream merge is directionally useful but must be handled as a dedicated merge cycle, not as a quick patch dependency for T306.

Upstream Merge Recon Implications

  • The merge verdict is a conditional yes, but the scale means it is a full dedicated upstream-merge cycle. T306 should not assume a quick merge or patch-on-current-fork path.
  • The stable upstream target is v2026.5.12 at 0358f3dda46dc17d3263e403f6ca895c4d0f749c; the fork baseline is origin/main at 6fbbb1188a71657e658639e981f0a08b8665dfe5.
  • The dry run found 38 conflict paths and roughly 8,967 staged changed paths. Product architecture should therefore specify the desired upstream-aligned contract first, then let the merge cycle decide whether the implementation lands by direct backport, post-merge adoption, or a compatibility shim.
  • For T306 direction, prefer upstream protocol v4 chat semantics: deltaText/replace content updates with seq, final, and error run framing, plus tool/progress event semantics for current-prompt activity. Task Ledger remains background-task support and is not enough alone for current-turn UX.

Background Tasks

OpenClaw has a separate background-task path. AgentInternalEventSchema includes task_completion events with sources such as subagent, cron, video generation, and music generation. There are also subagent registry and command surfaces that track active or linked task runs.

That path is valuable for Task Ledger and asynchronous task completion, but it is not the primary API for showing progress for the current user prompt. Mixing background task state into current-turn progress would create ambiguous UI: a user would not know whether a status line is about the prompt they just sent or a separate child/background job.

Current Clawline State

  • The Clawline provider dispatch path creates an activeSessionRuns entry and exposes it through /api/session-status. The status payload has run state, run ID, message ID, started time, queue depth, model/mode display, capabilities, and model catalog. It has no live item/plan/tool progress.
  • The provider emits only a coarse generic event with event: "activity" around dispatch. Native maps this to typing state.
  • The provider's createReplyDispatcherWithTyping usage does not attach the existing upstream progress callbacks for item, plan, approval, command-output, patch, compaction, or reasoning progress.
  • Native ProviderWireModels.swift has no typed live progress payload. ProviderChatService ignores unknown socket message types and only recognizes the generic activity event.
  • ChatServicing exposes message, service, and lifecycle streams, plus status fetching. It has no progress event stream.
  • ChatViewModel polls/schedules status refresh on selected session changes, auth changes, assistant commits, typing transitions, terminal events, and cancellation. That helps state convergence but is not live progress.
  • The message list footer renders session display/status controls. There is no current-turn progress lane. Existing typing UI can only show busy/typing, not what the agent is doing.
  • The client feature set currently advertises terminal_bubbles_v1; there is no live_agent_progress_v1 feature gate.

Important gap: upstream already has the progress information, but Clawline currently collapses it to a boolean activity signal plus a status snapshot. The minimum useful change is a projection/transport gap, not a Task Ledger feature.

Recommended Minimal Architecture

1. Provider Progress Projector

Add a small Clawline provider projector at the dispatch seam. It should consume upstream reply callbacks first because that is the seam the current Clawline provider already owns for inbound native messages. If/when the dedicated upstream merge lands protocol v4 in the fork, the same native protocol should be fed from protocol v4 chat framing plus tool/progress event frames.

The projector should normalize source events into user-safe progress events. It should not reconstruct layout or native presentation. The provider owns event projection; native owns rendering.

2. Additive Socket Protocol

Send additive websocket payloads to subscribed native sessions. Unknown types are already non-fatal in the current native client, so this can be introduced without disrupting older clients. Prefer a feature gate such as live_agent_progress_v1 before broad emission.

Align the content-message side with upstream protocol v4 where possible: assistant text should be represented as ordered deltaText/replace updates and closed by final or error. Progress should remain a sibling event stream, not a special assistant message.

{
  "type": "agent_progress",
  "version": 1,
  "sessionKey": "agent:main:clawline:flynn:main",
  "runId": "s_...",
  "messageId": "client-message-id",
  "seq": 12,
  "timestamp": 1778800000000,
  "state": "running",
  "event": {
    "kind": "item",
    "phase": "start",
    "status": "running",
    "title": "Reading files",
    "summary": "Reading files"
  }
}

Use a single event stream plus a native reducer rather than sending final UI layout. Optional reconnect support can add a volatile progress_snapshot later, either in auth_result/session_info or a new status-adjacent endpoint. The first implementation can fall back to /api/session-status showing "Running" after reconnect.

3. Projection Rules

Upstream Event Native Projection
Run seq/final/error framing Order current-run updates, clear progress on final, and show a failed/interrupted state on error.
Tool start/update/end Compact label plus optional safe tool name. Hide raw args by default.
Item start/update/end Primary current work item. Use progressText, title, or summary in that order.
Plan update Optional expanded progress details. Do not replace the transcript or assistant message stream.
Approval requested/resolved Show waiting/blocked state tied to the current run. Do not invent approval controls beyond existing approved flows.
Command output Show command running/finished state. Do not stream raw output in compact mode.
Patch summary Show concise file-change summary if safe for the session.
Compaction Show a short pause/progress label only while active.
Assistant deltaText/replace Keep in message streaming. Do not duplicate as progress.

4. Native Service and State

  • Add AgentProgressEvent to the wire model, and keep it sibling to any protocol v4 message delta/final/error models.
  • Add a progress async stream or service event in ChatServicing/ProviderChatService.
  • Reduce events in ChatViewModel by sessionKey and runId. Use upstream seq defensively for both content and progress; tolerate gaps, repeats, and stale events.
  • Clear progress on assistant final, lifecycle end/error, cancellation, selected-session teardown, or stale timeout.
  • Do not persist volatile progress as chat history and do not advance replay cursors from progress events.

5. Native UI

Start with one compact current-turn surface for the selected stream: a work strip or enriched typing/progress indicator near the message flow/composer. Show one current label and, if expanded, the latest few items. The session footer should continue to own model/status/control metadata, not step-by-step progress.

What To Implement First

  1. Define the native contract around upstream protocol v4 semantics: ordered chat deltaText/replace plus final/error, and sibling tool/progress events for current-prompt activity.
  2. Add provider-side live progress projection from the Clawline dispatch callbacks into an additive agent_progress websocket event while the full upstream merge cycle remains separate.
  3. Add native decode, service stream, reducer, and a minimal current-turn UI surface.
  4. Add focused tests: provider mapping tests, Swift decode/reducer tests, and one UI behavior test showing progress appears during a run and clears on final/error.
  5. Add reconnect fallback using current /api/session-status. Only add a retained progress snapshot if testing shows reconnect UX is too thin.

Do not start with Task Ledger. Do not start by expanding SessionStatus into a progress timeline. Status/control is a snapshot API; current-turn progress is a live event stream. Do not make T306 contingent on completing the 38-conflict-path upstream merge, but keep the protocol pointed at that upstream destination.

Risks

  • Event ordering and loss: upstream live events may be dropped for slow subscribers or missed by late clients. Native must treat progress as best-effort and reconstruct only current display state, not authoritative history.
  • Merge scale: the upstream merge is conditional yes but large: 38 conflict paths and about 8,967 staged changed paths. Treat merge sequencing as a project risk, not an implementation detail for a quick T306 patch.
  • Privacy and command detail: raw tool args, command output, paths, or approval reasons can expose sensitive data. Default to compact sanitized labels and make raw detail an explicit later decision.
  • Protocol drift: local Gateway broadcasts include session routing fields, while the agent schema may lag what is actually sent. Protocol v4 adoption should be explicit and tested rather than inferred from current fork behavior.
  • Transcript pollution: progress must never be stored as assistant messages or replayed as chat history.
  • Double rendering: assistant text streaming and progress status must remain separate. The Codex app-server path intentionally suppresses intermediate assistant messages until final selection, so progress should describe work, not partial answers.
  • Multi-device routing: session key subscriptions remain the routing truth. The bridge must only emit to provisioned/adopted Clawline streams that are subscribed to the target session.
  • Runtime variability: Codex app-server emits rich item/plan events; other agent runtimes may only provide lifecycle/tool/assistant signals. The UI must handle sparse streams gracefully.
  • Feature negotiation: older clients ignore unknown types today, but a feature flag still reduces log noise and lets the provider avoid unnecessary emissions.

Open Questions

  • Should native initially receive only compact labels, or should it also receive safe structured details for a future expanded view?
  • Should reconnect snapshot state be in-memory only, or should the provider expose a small volatile endpoint? Recommendation: in-memory only, if needed after first UI testing.
  • Should the same agent_progress protocol be wired into the web Clawline client immediately, or should native ship first and web follow once the protocol stabilizes?
  • Should T306 land as a compatibility bridge before the full upstream merge, or wait for the dedicated upstream merge cycle and then implement directly against protocol v4? Recommendation: define the protocol now; implementation sequencing should follow merge ownership and schedule.
  • Should reasoning progress be hidden entirely by default? Recommendation: do not show raw reasoning; use explicit item/tool/plan labels first.

Architecture Boundary

The provider should emit resolved progress facts, not UI layout intent. Native presentation owns placement, truncation, expansion, animation, and accessibility. This keeps one source of truth for event semantics and avoids leaking provider/channel rendering decisions into the app.

Conclusion

Clawline does not need Task Ledger as the primary current-turn progress API. The upstream system already has live per-run progress via protocol v4 chat framing, tool/progress events, agent events, and reply callbacks. The lowest-risk product path is an additive progress event from the Clawline provider, a native reducer keyed by session/run/sequence, and a compact UI surface that clears on final or shows failure on error. The full upstream merge remains a separate dedicated cycle.
