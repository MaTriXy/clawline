T314 Vim Mode Next-Step Suggestions

Product and architecture specification for local Foundation Model next-step suggestions over visible Clawline assistant-message bubbles. Prepared 2026-05-16; revised 2026-05-17.

Decision: Cmd-F enters transient vim mode over currently visible assistant-message bubbles only. Lettered suggestions render near their source bubbles, typing a valid assigned letter immediately hands that suggestion prompt to the normal current-chat send path and exits vim mode, and Esc exits without side effects.

Vim mode is a keyboard-only "vim hint" layer. It snapshots visible assistant-message bubble content and resolved bubble rectangles, asks the platform-local Foundation Model for concise next-step prompts, assigns unique letters, and renders non-persistent overlays. No suggestion context leaves the device before the user selects a suggestion.

Goals

  • Let a physical-keyboard user press Cmd-F from chat transcript context to enter vim mode for local next-step suggestions over assistant-message bubbles that are currently visible in the active chat.
  • Use the platform-local Foundation Model to generate concise prompt suggestions from visible assistant-message bubble text only.
  • Render each suggestion as a compact lettered hint near the assistant-message bubble that produced it.
  • Submit the selected suggestion by typing its assigned letter, using the same prompt submission route as a normal current-chat prompt.
  • Exit cleanly on Esc, navigation, unsupported local model state, timeout/error, no valid candidates, or source-visibility invalidation.

Non-Goals

  • No remote model calls, provider-side suggestion generation, analytics upload, cross-device sync, or persistent suggestion history.
  • No suggestions from user, system, tool, terminal, status, typing, progress, composer, notification, hidden, offscreen, or previously visible bubbles.
  • No transcript mutation, draft insertion, or composer text replacement until the user types a valid suggestion letter.
  • No multi-letter chords, tap/click selection, touch-only entry UI, mouse-only selection requirement, or suggestion editing surface in this v1 feature.
  • No new routing contract. Selected suggestions submit through the existing current-chat prompt pathway.
  • No platform fallback that uses a non-local model when the local Foundation Model is unavailable.

Mode Lifecycle

  1. User presses Cmd-F while the active chat transcript can receive global chat shortcuts. If focus is in a composer or any text-editing field, native Cmd-F behavior for that field wins and vim mode is not requested.
  2. If vim mode is already active, Cmd-F is ignored for v1. It does not regenerate, cancel/restart, submit, or change the active hint set.
  3. The chat screen captures a snapshot of eligible visible assistant-message bubbles: stable message identity, bounded rendered visible text excerpt, and resolved bubble rectangle in the owning screen coordinate space.
  4. If the snapshot has no eligible sources, Clawline does not call the local model and does not enter vim mode.
  5. Before generation, Clawline checks SystemLanguageModel.default.availability. If the local model is unavailable, disabled, downloading, rate-limited, or otherwise not ready, Clawline does not enter vim mode and performs no remote fallback.
  6. The local suggestion service creates a fresh LanguageModelSession for each vim-mode request. The request receives only the bounded captured text excerpts and message identifiers needed to associate results back to source bubbles.
  7. Generation runs as bounded, cancellable async work outside cell configuration. Cell configure/reuse may expose already-rendered text and resolved rectangles to the snapshot path, but it must never start Foundation Model generation.
  8. When local generation returns valid candidates, the reducer filters invalid, empty, duplicate, and unassociated suggestions, assigns unique letters, enters active vim mode, and renders hints near source bubbles.
  9. If generation errors, times out, is cancelled, or returns no valid candidates, Clawline remains out of vim mode or exits the request path without transcript/provider/gateway/analytics/replay/notification-store writes.
  10. Typing a valid assigned letter immediately hands that suggestion as the next prompt to the current chat send pipeline and exits vim mode immediately. Send acceptance or failure is then handled only by the normal send UI.
  11. Esc, chat navigation, source bubble disappearance, or local-generation failure exits vim mode without submitting anything.

Eligibility

Eligible Sources

  • Only assistant-message transcript bubbles in the current chat are eligible.
  • A bubble is eligible only if it is currently visible in the active transcript viewport when Cmd-F is pressed.
  • Visibility is determined from resolved rectangles owned by the chat screen or message list layout. The suggestion overlay receives resolved source rectangles and must not reconstruct scroll position, split orientation, pane topology, or message layout intent.
  • Streaming assistant-message bubbles are eligible only when they already expose visible assistant text. Empty typing indicators, progress strips, and placeholders are not eligible.
  • Assistant-message bubbles with no text extractable for local generation are not eligible. Attachments or rich content may contribute only text already exposed by the existing transcript preview/extraction policy.

Ineligible Sources

  • User, system, tool, terminal, status, typing, progress, composer, notification, cross-chat overlay, and any other non-assistant-message surfaces are ineligible.
  • Assistant-message bubbles outside the current viewport are ineligible even if they are in the current transcript data source.
  • Assistant-message bubbles from other chats are ineligible. This feature does not inspect hidden chats or cross-chat notification state.
  • Bubbles hidden by navigation, filtering, redaction, modal presentation, or cell reuse at mode entry are ineligible.

Snapshot Semantics

  • Eligibility is snapshot-based. Scrolling after mode entry does not add new sources or remove generated suggestions by itself.
  • If the source bubble set is structurally invalidated by chat navigation, transcript reload, message deletion, or selected-session change, vim mode exits.
  • If an eligible streaming assistant-message bubble receives more text after the snapshot, the active suggestion request does not incorporate the new text. Because Cmd-F is ignored while vim mode is active in v1, the user must exit and re-enter mode to request fresh suggestions.

Local Foundation Model and Privacy

  • Suggestion generation runs only through the platform-local Foundation Model facility available to the Clawline client process.
  • Availability is checked with SystemLanguageModel.default.availability before issuing generation. Any unavailable, disabled, downloading, rate-limited, unsupported, timeout, cancellation, or error state keeps vim mode out or exits it with no side effects.
  • Each vim-mode request uses a fresh LanguageModelSession. Clawline does not reuse a session for later vim-mode requests and does not maintain a persistent suggestion cache.
  • The provider, gateway, OpenClaw runtime, remote model APIs, analytics, replay, transcript persistence, and notification-store paths do not receive source-bubble text, prompt templates, suggestion candidates, mode telemetry, failure telemetry, or generation state before the user selects a suggestion.
  • The local model prompt includes only bounded rendered visible assistant-message bubble text excerpts required for current suggestion generation, the source message identifiers needed for local association, and the instruction to produce concise next-step prompts.
  • Suggestions are volatile UI state. They are not written to transcript storage, replay logs, provider status, analytics, cross-chat notification stores, or sync state.
  • When the user types a valid suggestion letter, only the selected suggestion text is submitted through the normal current-chat prompt pathway. At that point it has the same privacy and routing semantics as a user-authored prompt.
  • If the local Foundation Model is unavailable, disabled, downloading, rate-limited, times out, is cancelled, or returns an error, Clawline exits or stays out of vim mode with no side effects. It must not fall back to a remote model.

Suggestion Generation Contract

  • The local generation request groups text by source bubble. The model should produce zero or more concise next-step prompt candidates per source bubble.
  • Each suggestion should be written as a prompt the user could send directly, not as an explanation of why the suggestion exists.
  • Suggestions should be short enough to scan in an overlay. Target one line; allow wrapping only when the platform cannot fit a complete useful prompt otherwise.
  • Suggestions must preserve source association. If the model cannot associate a candidate to a source bubble, discard that candidate.
  • Duplicate or near-duplicate candidates in the same active mode are collapsed before letter assignment. Empty candidates and candidates associated with no source bubble are discarded.
  • The prompt input is bounded rendered visible text from the snapshot, not the full transcript, hidden cell content, provider payloads, message metadata, or offscreen transcript state.
  • Generation work is cancellable and bounded by an implementation-defined timeout. Cancellation, timeout, or error discards candidates and performs no writes.
  • Generation must never be started from cell configure/reuse. The cell path may participate only by exposing already-rendered visible text and resolved rectangles to the snapshot owner.
  • The generation layer returns suggested prompt text only. It does not return overlay coordinates, typography, animation, hit regions, routing destinations, or submission commands.

Suggestion Placement

  • The owning chat screen resolves source bubble rectangles and viewport bounds. The suggestion overlay receives those resolved rectangles.
  • Each hint renders visually near its source bubble, preferring the trailing edge of the assistant-message bubble on wide layouts and a safe below-or-above edge on compact layouts.
  • Hints must stay within the resolved visible viewport and safe-area bounds. If preferred placement would clip, the overlay may flip to the other side of the source bubble or stack hints along the nearest visible edge.
  • Multiple hints for one source bubble stack together in source order with stable spacing and must not cover the letter labels of other hints.
  • Hints from adjacent bubbles may compact vertically but must preserve clear source association. A connector line is optional; proximity and ordering must be sufficient without requiring one.
  • The overlay must not move transcript cells, resize bubbles, alter scroll position, or reserve layout space. It is a presentation layer above the transcript.
  • Tap/click selection is not part of v1. Visible hints are keyboard-selected only by assigned letters.

Geometry boundary: message layout owns source rectangles. The suggestion overlay may validate bounds, transform coordinate spaces, place hints, and route hit regions against provided rectangles. It must not infer pane topology, split direction, orientation, scroll math, or transcript layout intent.

Unique-Letter Assignment

  • Letters are unique within one active vim mode.
  • The assignment pool is lowercase a through z. The display may render uppercase letter badges if that matches platform style, but typed lowercase and uppercase forms select the same hint.
  • Assignment order follows visible source bubble order from top to bottom, then suggestion order within each source bubble.
  • If more than 26 valid suggestions are generated, only the first 26 by assignment order are shown. Extra candidates are discarded for that active mode.
  • Letters are frozen after assignment. They are not reassigned while the same vim mode remains active, even if overlays animate or the transcript receives streaming updates.
  • Invalid candidates, empty candidates, duplicates, and candidates whose source bubble is no longer in the snapshot are removed before letters are assigned.

Focus and Keyboard Semantics

Entering Mode

  • Cmd-F enters vim mode only when the current chat transcript is active and focus is not inside a text-editing field.
  • If focus is inside the composer, a notification reply field, a search field, or another text editor, Cmd-F keeps the existing native text-editing meaning for that field and does not enter vim mode.
  • If a modal, sheet, confirmation dialog, system permission prompt, or blocking overlay is active, Cmd-F does not enter vim mode.
  • Re-pressing Cmd-F while vim mode is already active is ignored for v1. It must not regenerate suggestions, cancel/restart the request, clear active hints, or submit anything.

Active Mode

  • While active, vim mode captures unmodified letter keys that match assigned letters.
  • Typing an assigned letter immediately hands the associated suggestion prompt to the existing current-chat send pipeline and exits vim mode immediately. Send acceptance/failure is handled by the normal send UI after vim mode exits.
  • Typing an unassigned letter is a no-op and keeps vim mode active.
  • Esc exits vim mode before transcript/chat scroll handling. It does not submit, change composer text, or mutate transcript state.
  • Pointer/touch selection of hints is intentionally absent in v1.
  • Existing global shortcuts not listed here should either keep their existing app-level behavior if they do not conflict with mode capture, or be ignored while the mode is active. They must not submit a suggestion unless they resolve to an assigned letter selection.
  • Text input focus should return to the same chat-level focus owner that was active before mode entry unless a valid letter selection has handed off to the normal prompt submission flow.

Prompt Submission Routing

  • A selected suggestion submits as a user prompt to the current chat session, using the existing current-chat prompt submission route.
  • The selected suggestion is not submitted to the source bubble's historical run, another chat, the provider as metadata, or any special suggestion endpoint.
  • The resulting user message appears in the current chat transcript exactly as a normal submitted prompt would.
  • Current chat dot/status, send state, queueing, cancellation, and failure behavior are owned by the existing send pipeline.
  • If the user navigates to another chat before pressing a letter, vim mode exits. A letter typed after navigation cannot submit against the previous chat snapshot.
  • If the normal send pipeline rejects or fails the selected suggestion after handoff, the existing prompt send-error pathway handles the failure. Vim mode has already exited and must not create a separate transcript row or retry path.

Failure States

State Required Behavior
No eligible visible assistant-message bubbles Do not call the local model. Remain out of vim mode with no side effects and no transcript/provider/gateway/analytics/replay/notification-store writes.
Local Foundation Model unavailable Do not fall back to remote generation. Remain out of or exit vim mode with no side effects and no writes.
Local generation times out, is cancelled, or errors Clear pending UI, discard partial candidates, and remain out of or exit vim mode without transcript/provider/gateway/analytics/replay/notification-store writes.
Local generation returns no valid candidates Render no hints and remain out of or exit vim mode with no side effects. No remote fallback and no writes.
Source snapshot invalidated Discard active suggestions and exit mode.
Assigned letter selected while send is unavailable Hand the selected prompt to the existing send pipeline, exit vim mode immediately, and display whatever existing send-unavailable failure state the send pipeline produces. Do not invent a second failure system.

Platform Differences

  • On macOS and Catalyst, Cmd-F is the primary entry shortcut when focus is in the chat transcript and not in text editing. The feature intentionally uses the app-level chat shortcut in that focus state rather than opening global text find.
  • On iPadOS with a hardware keyboard, Cmd-F, letter selection, and Esc follow the same semantics as macOS/Catalyst.
  • On iPhone or iPad without a hardware keyboard, this spec does not require a touch-only entry surface. Existing UI may omit the feature or expose it later through a separate spec.
  • On platforms without the required local Foundation Model API, vim mode is unavailable. Web/Electron must not use a remote substitute under this spec.
  • Hints use native platform typography and materials where available. Apple platforms should use SF fonts and platform materials consistent with the surrounding Clawline UI.
  • Accessibility labels for hints identify the assigned letter and suggestion text. v1 does not add pointer/touch activation or a separate accessibility activation route; selection is by keyboard letter.

Proof Gates

Eligibility Checks

  • Given mixed visible transcript items, only visible assistant-message bubbles become suggestion sources.
  • Given hidden/offscreen assistant-message bubbles, no source is created for them.
  • Given user, system, tool, terminal, status, typing, progress, composer, notification, or other non-assistant-message surfaces, no source is created for them.
  • Given no eligible visible assistant-message bubbles, the local model is not called and vim mode is not entered.

Reducer and Letter Checks

  • Given duplicate, empty, or unassociated model candidates, the reducer discards them before letter assignment.
  • Given more than 26 candidates, only 26 hints render and each has a unique letter.
  • Given an assigned letter, the reducer hands exactly that prompt to the existing current-chat send pipeline and exits vim mode immediately.
  • Given an unassigned letter, the reducer performs no send action and keeps vim mode active.
  • Given a snapshot invalidation event, vim mode exits and clears hints.

Local Model and Privacy Checks

  • Availability is checked through SystemLanguageModel.default.availability before generation.
  • Each vim-mode request creates a fresh LanguageModelSession; no persistent suggestion cache is read or written.
  • The generation request includes only bounded rendered visible text from eligible assistant-message bubbles and the minimum local association identifiers.
  • Generation runs as bounded, cancellable async work and is never started from cell configure/reuse.
  • Instrumentation or tests verify that entering, generating, failing, timing out, or producing no candidates performs no provider, gateway, analytics, replay, notification-store, or remote model request/write.
  • Only the selected suggestion text is sent to the provider, and only after the user types an assigned letter and the existing send pipeline owns the prompt.
  • Unselected suggestions are absent from transcript storage, replay logs, provider status, analytics, notification state, and sync state.

Keyboard and Routing Checks

  • Cmd-F enters vim mode from chat transcript focus and does not enter vim mode from composer/text-field focus, where native Cmd-F remains available.
  • Cmd-F while vim mode is active is ignored and does not regenerate.
  • Esc exits vim mode before transcript/chat scroll handling, without changing composer text or transcript state.
  • Typing an assigned letter submits exactly that hint's prompt through the normal current-chat send route and exits vim mode immediately.
  • Typing an unassigned letter does not submit and keeps mode active.
  • Navigating chats before letter selection exits mode and prevents stale snapshot submission.
  • Tap/click selection is absent for v1.

Placement Checks

  • Hints render near their source bubbles on compact iPhone, iPad split view with hardware keyboard, macOS/Catalyst, and any supported web/Electron surface that has local-model support.
  • Hints stay within safe visible bounds and do not move transcript layout or scroll position.
  • Adjacent source bubbles with multiple hints remain readable and preserve source association.
  • The overlay consumes resolved source rectangles and does not reconstruct pane topology, split direction, orientation, scroll math, or transcript layout intent.

Integration and Build Checks

  • The implementation builds successfully for the target Clawline app configuration before handoff.
  • Automated or instrumented checks prove no provider/gateway/transcript/analytics/replay/notification-store write occurs on vim-mode entry, generation, failure, timeout, cancellation, or no-candidate results.
  • Existing current-chat send UI owns send acceptance, send failure, queueing, and unavailable states after assigned-letter handoff.

Simulator or Manual Proof

  • At least one simulator or manual run demonstrates visible assistant-only suggestions: visible assistant-message bubbles receive hints, while visible user/system/tool/terminal/status/typing/progress/composer/notification surfaces do not.
  • The same proof demonstrates assigned-letter submission into the existing current-chat send pipeline and immediate vim-mode exit.
  • The proof records the environment, visible source setup, assigned letter pressed, and resulting normal send UI behavior.

Implementation Handoff

  • Implement only the local vim mode described here: Cmd-F entry from transcript/chat context, visible assistant-message bubble snapshot, local Foundation Model generation, unique lettered overlays, assigned-letter submission, ignored active-mode Cmd-F, and Esc exit.
  • Use SystemLanguageModel.default.availability, a fresh LanguageModelSession per vim-mode request, bounded rendered visible text, no persistent suggestion cache, bounded/cancellable async work, and no generation from cell configure/reuse.
  • Do not add remote fallback, persistent suggestion history, tap/click selection, touch-only entry UI, cross-chat suggestions, transcript annotations, extra routing APIs, analytics/replay/notification-store writes, or provider-side generation unless this spec is updated first.
  • Respect the geometry boundary: the chat/message layout layer owns resolved source rectangles; the suggestion overlay consumes resolved rectangles and owns only presentation and hit regions.
  • Use the existing current-chat prompt submission route. Do not create a special suggestion send path, special send failure path, or delayed vim-mode exit tied to send acceptance.
  • Implementation discovery still needs to identify the existing transcript rendered-text extraction helper and the exact current-chat send entry point to reuse. If either reuse point is ambiguous, ask for spec clarification rather than inventing a parallel path.
