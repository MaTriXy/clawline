T314 Vim-Mode Next-Step Suggestions

Product and architecture specification for local Foundation Model next-step suggestions over visible Clawline assistant bubbles. Prepared 2026-05-16.

Decision: Cmd-F enters a transient local suggestion mode over currently visible assistant bubbles only. Lettered suggestions render near their source bubbles, typing a suggestion letter submits that suggestion through the normal current-chat prompt path, and Esc exits without side effects.

The mode is a keyboard-first "vim hint" layer. It snapshots visible assistant bubble content and resolved bubble rectangles, asks the platform-local Foundation Model for concise next-step prompts, assigns unique letters, and renders non-persistent overlays. No suggestion context leaves the device before the user selects a suggestion.

Goals

  • Let a physical-keyboard user press Cmd-F from the chat transcript to request local next-step suggestions for assistant bubbles that are currently visible in the active chat.
  • Use the platform-local Foundation Model to generate concise prompt suggestions from visible assistant-bubble text only.
  • Render each suggestion as a compact lettered hint near the assistant bubble that produced it.
  • Submit the selected suggestion by typing its assigned letter, using the same prompt submission route as a normal current-chat prompt.
  • Exit cleanly on Esc, navigation, unsupported local model state, or source-visibility invalidation.

Non-Goals

  • No remote model calls, provider-side suggestion generation, analytics upload, cross-device sync, or persistent suggestion history.
  • No suggestions from user, system, tool, terminal, notification, hidden, offscreen, or previously visible bubbles.
  • No transcript mutation, draft insertion, or composer text replacement until the user types a valid suggestion letter.
  • No multi-letter chords, mouse-only selection requirement, or suggestion editing surface in this feature.
  • No new routing contract. Selected suggestions submit through the existing current-chat prompt pathway.
  • No platform fallback that uses a non-local model when the local Foundation Model is unavailable.

Terminology

Term Meaning
Suggestion mode The transient keyboard-capture state entered by Cmd-F and exited by Esc or mode-ending events.
Source bubble One eligible assistant transcript bubble that was visible when suggestion mode entered.
Visible assistant bubble An assistant-role transcript bubble whose resolved cell/content rectangle intersects the active transcript viewport at mode entry and is not hidden behind another presentation surface.
Suggestion hint A compact overlay containing one unique letter and one concise prompt suggestion.
Prompt submission route The same current-chat send path used when the user types text into the composer and submits it normally.

Mode Lifecycle

  1. User presses Cmd-F while the active chat transcript can receive global chat shortcuts.
  2. The chat screen captures a snapshot of eligible visible assistant bubbles: stable message identity, rendered text excerpt, and resolved bubble rectangle in the owning screen coordinate space.
  3. The local suggestion service receives only the captured text excerpts and message identifiers needed to associate results back to source bubbles.
  4. While generation is pending, the UI enters suggestion mode and may show a small local-generation pending marker near eligible bubbles or the transcript edge.
  5. When local generation returns, the reducer filters invalid or empty suggestions, assigns unique letters, and renders hints near source bubbles.
  6. Typing a valid assigned letter submits that suggestion as the next prompt to the current chat and exits suggestion mode after the send is accepted by the normal submission path.
  7. Esc, chat navigation, source bubble disappearance, or local-generation failure exits suggestion mode without submitting anything.

Eligibility

Eligible Sources

  • Only assistant-role transcript bubbles in the current chat are eligible.
  • A bubble is eligible only if it is currently visible in the active transcript viewport when Cmd-F is pressed.
  • Visibility is determined from resolved rectangles owned by the chat screen or message list layout. The suggestion overlay receives resolved source rectangles and must not reconstruct scroll position, split orientation, pane topology, or message layout intent.
  • Streaming assistant bubbles are eligible only when they already expose visible assistant text. Empty typing indicators, progress strips, and placeholders are not eligible.
  • Assistant bubbles with no text extractable for local generation are not eligible. Attachments or rich content may contribute only text already exposed by the existing transcript preview/extraction policy.

Ineligible Sources

  • User, system, tool, terminal, status, notification, cross-chat overlay, composer, and progress surfaces are ineligible.
  • Assistant bubbles outside the current viewport are ineligible even if they are in the current transcript data source.
  • Assistant bubbles from other chats are ineligible. This feature does not inspect hidden chats or cross-chat notification state.
  • Bubbles hidden by navigation, filtering, redaction, modal presentation, or cell reuse at mode entry are ineligible.

Snapshot Semantics

  • Eligibility is snapshot-based. Scrolling after mode entry does not add new sources or remove generated suggestions by itself.
  • If the source bubble set is structurally invalidated by chat navigation, transcript reload, message deletion, or selected-session change, suggestion mode exits.
  • If an eligible streaming assistant bubble receives more text after the snapshot, the active suggestion request does not incorporate the new text. The user can exit and re-enter mode to regenerate suggestions.

Local Foundation Model and Privacy

  • Suggestion generation runs only through the platform-local Foundation Model facility available to the Clawline client process.
  • The provider, gateway, OpenClaw runtime, and remote model APIs do not receive source-bubble text, prompt templates, suggestion candidates, or mode telemetry before the user selects a suggestion.
  • The local model prompt includes only the visible assistant-bubble text excerpts required for current suggestion generation, the source message identifiers needed for local association, and the instruction to produce concise next-step prompts.
  • Suggestions are volatile UI state. They are not written to transcript storage, replay logs, provider status, cross-chat notification stores, or sync state.
  • When the user types a valid suggestion letter, only the selected suggestion text is submitted through the normal current-chat prompt pathway. At that point it has the same privacy and routing semantics as a user-authored prompt.
  • If the local Foundation Model is unavailable, disabled, downloading, rate-limited, or returns an error, Clawline exits or stays out of suggestion mode with a compact local-only failure indication. It must not fall back to a remote model.

Suggestion Generation Contract

  • The local generation request groups text by source bubble. The model should produce zero or more concise next-step prompt candidates per source bubble.
  • Each suggestion should be written as a prompt the user could send directly, not as an explanation of why the suggestion exists.
  • Suggestions should be short enough to scan in an overlay. Target one line; allow wrapping only when the platform cannot fit a complete useful prompt otherwise.
  • Suggestions must preserve source association. If the model cannot associate a candidate to a source bubble, discard that candidate.
  • Duplicate or near-duplicate candidates in the same active mode are collapsed before letter assignment.
  • The generation layer returns suggested prompt text only. It does not return overlay coordinates, typography, animation, hit regions, routing destinations, or submission commands.

Suggestion Placement

  • The owning chat screen resolves source bubble rectangles and viewport bounds. The suggestion overlay receives those resolved rectangles.
  • Each hint renders visually near its source bubble, preferring the trailing edge of the assistant bubble on wide layouts and a safe below-or-above edge on compact layouts.
  • Hints must stay within the resolved visible viewport and safe-area bounds. If preferred placement would clip, the overlay may flip to the other side of the source bubble or stack hints along the nearest visible edge.
  • Multiple hints for one source bubble stack together in source order with stable spacing and must not cover the letter labels of other hints.
  • Hints from adjacent bubbles may compact vertically but must preserve clear source association. A connector line is optional; proximity and ordering must be sufficient without requiring one.
  • The overlay must not move transcript cells, resize bubbles, alter scroll position, or reserve layout space. It is a presentation layer above the transcript.
  • The overlay may route taps/clicks on visible hints as an optional equivalent to typing the assigned letter, but keyboard letter selection remains the primary interaction.

Geometry boundary: message layout owns source rectangles. The suggestion overlay may validate bounds, transform coordinate spaces, place hints, and route hit regions against provided rectangles. It must not infer pane topology, split direction, orientation, scroll math, or transcript layout intent.

Unique-Letter Assignment

  • Letters are unique within one active suggestion mode.
  • The assignment pool is lowercase a through z. The display may render uppercase letter badges if that matches platform style, but typed lowercase and uppercase forms select the same hint.
  • Assignment order follows visible source bubble order from top to bottom, then suggestion order within each source bubble.
  • If more than 26 valid suggestions are generated, only the first 26 by assignment order are shown. Extra candidates are discarded for that active mode.
  • Letters are frozen after assignment. They are not reassigned while the same suggestion mode remains active, even if overlays animate or the transcript receives streaming updates.
  • Invalid candidates, empty candidates, duplicates, and candidates whose source bubble is no longer in the snapshot are removed before letters are assigned.

Focus and Keyboard Semantics

Entering Mode

  • Cmd-F enters suggestion mode only when the current chat transcript is active and focus is not inside a text-editing field.
  • If focus is inside the composer, a notification reply field, a search field, or another text editor, Cmd-F keeps the existing text-editing meaning for that field and does not enter suggestion mode.
  • If a modal, sheet, confirmation dialog, system permission prompt, or blocking overlay is active, Cmd-F does not enter suggestion mode.
  • Re-pressing Cmd-F while suggestion mode is already active regenerates suggestions from a fresh visible-bubble snapshot only after first canceling the active mode state.

Active Mode

  • While active, suggestion mode captures unmodified letter keys that match assigned letters.
  • Typing an assigned letter submits the associated suggestion. Typing an unassigned letter is a no-op and keeps suggestion mode active.
  • Esc exits suggestion mode without submitting, without changing composer text, and without mutating transcript state.
  • Pointer/touch selection of a hint, if implemented, must behave exactly like typing that hint's assigned letter.
  • Existing global shortcuts not listed here should either keep their existing app-level behavior if they do not conflict with mode capture, or be ignored while the mode is active. They must not submit a suggestion unless they resolve to an assigned letter selection.
  • Text input focus should return to the same chat-level focus owner that was active before mode entry unless a valid letter selection has handed off to the normal prompt submission flow.

Prompt Submission Routing

  • A selected suggestion submits as a user prompt to the current chat session, using the existing current-chat prompt submission route.
  • The selected suggestion is not submitted to the source bubble's historical run, another chat, the provider as metadata, or any special suggestion endpoint.
  • The resulting user message appears in the current chat transcript exactly as a normal submitted prompt would.
  • Current chat dot/status, send state, queueing, cancellation, and failure behavior are owned by the existing send pipeline.
  • If the user navigates to another chat before pressing a letter, suggestion mode exits. A letter typed after navigation cannot submit against the previous chat snapshot.
  • If the normal send pipeline rejects or fails the selected suggestion, the existing prompt send-error pathway handles the failure. Suggestion mode must not create a separate transcript row or retry path.

Failure States

State Required Behavior
No eligible visible assistant bubbles Do not call the local model. Show a compact no-suggestions indication or simply remain out of suggestion mode.
Local Foundation Model unavailable Do not fall back to remote generation. Show a compact local-model-unavailable indication and exit suggestion mode.
Local generation times out, is cancelled, or errors Clear pending UI, discard partial candidates, and exit suggestion mode without transcript mutation.
Local generation returns no valid candidates Show no hints and exit suggestion mode or show a compact no-suggestions indication. No remote fallback.
Source snapshot invalidated Discard active suggestions and exit mode.
Assigned letter selected while send is unavailable Hand the selected prompt to the existing send pipeline and display whatever existing send-unavailable failure state it produces. Do not invent a second failure system.

Platform Differences

  • On macOS and Catalyst, Cmd-F is the primary entry shortcut when focus is in the chat transcript and not in text editing. The feature intentionally uses the app-level chat shortcut in that focus state rather than opening global text find.
  • On iPadOS with a hardware keyboard, Cmd-F, letter selection, and Esc follow the same semantics as macOS/Catalyst.
  • On iPhone or iPad without a hardware keyboard, this spec does not require a touch-only entry surface. Existing UI may omit the feature or expose it later through a separate spec.
  • On platforms without the required local Foundation Model API, suggestion mode is unavailable. Web/Electron must not use a remote substitute under this spec.
  • Hints use native platform typography and materials where available. Apple platforms should use SF fonts and platform materials consistent with the surrounding Clawline UI.
  • Accessibility labels for hints identify the assigned letter and suggestion text. Accessibility actions, if added, must trigger the same selection route as the assigned letter.

Proof Gates

Unit and Reducer Checks

  • Given mixed visible transcript items, only visible assistant bubbles become suggestion sources.
  • Given hidden/offscreen assistant bubbles, no source is created for them.
  • Given duplicate, empty, or unassociated model candidates, the reducer discards them before letter assignment.
  • Given more than 26 candidates, only 26 hints render and each has a unique letter.
  • Given a snapshot invalidation event, suggestion mode exits and clears hints.

Privacy Checks

  • Instrumentation or tests verify that entering suggestion mode performs no provider, gateway, or remote model request.
  • Only the selected suggestion text is sent to the provider, and only after the user types an assigned letter or activates the equivalent hint action.
  • Unselected suggestions are absent from transcript storage, replay logs, provider status, and notification state.

Keyboard and Routing Checks

  • Cmd-F enters mode from chat transcript focus and does not enter mode from composer/text-field focus.
  • Esc exits without changing composer text or transcript state.
  • Typing an assigned letter submits exactly that hint's prompt through the normal current-chat send route.
  • Typing an unassigned letter does not submit and keeps mode active.
  • Navigating chats before letter selection exits mode and prevents stale snapshot submission.

Placement Checks

  • Hints render near their source bubbles on compact iPhone, iPad split view with hardware keyboard, macOS/Catalyst, and any supported web/Electron surface that has local-model support.
  • Hints stay within safe visible bounds and do not move transcript layout or scroll position.
  • Adjacent source bubbles with multiple hints remain readable and preserve source association.

Implementation Handoff

  • Implement only the local suggestion mode described here: Cmd-F entry, visible assistant bubble snapshot, local Foundation Model generation, unique lettered overlays, letter selection submission, and Esc exit.
  • Do not add remote fallback, persistent suggestion history, touch-only entry UI, cross-chat suggestions, transcript annotations, extra routing APIs, or provider-side generation unless this spec is updated first.
  • Respect the geometry boundary: the chat/message layout layer owns resolved source rectangles; the suggestion overlay consumes resolved rectangles and owns only presentation and hit regions.
  • Use the existing current-chat prompt submission route. Do not create a special suggestion send path.
  • Open question for implementation discovery: identify the exact local Foundation Model availability API and the existing transcript text-extraction helper to reuse. This spec requires local-only generation and visible-bubble text extraction, but does not prescribe concrete type names.
