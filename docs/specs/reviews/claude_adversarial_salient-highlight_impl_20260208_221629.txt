### 2. **Race condition: `inFlight` dictionary mutation outside atomicity** (Concurrency bug)

In `Worker.generate()`:

```swift
let task = Task { [analysisText, renderedTextForResolution] in
    await self.performGeneration(...)
}

inFlight[key] = task

inFlight.removeValue(forKey: key)  // <-- Race here
```

Between `await task.value` returning and `removeValue`, another caller could:
1. Check `inFlight[key]` — finds the old task
2. Await it — gets the result
3. Then this line removes it
4. A *third* caller now finds no entry and starts a duplicate generation

This is a classic check-then-act race. The `inFlight` cleanup should happen atomically with result caching, or use a different pattern (e.g., store `Task<SalientHighlights?, Never>?` and nil it before awaiting, or use a continuation-based coalescing pattern).

### 3. **`LRUCache.get()` is O(n) — performance risk during scroll**

```swift
func get(_ key: Key) -> Value? {
    lock.lock()
    defer { lock.unlock() }
    guard let value = dict[key] else { return nil }
    if let idx = order.firstIndex(of: key) {  // O(n)
        order.remove(at: idx)                   // O(n)
        order.append(key)
    }
    return value
}
```

With `maxEntries = 600`, every cache hit does two O(n) operations while holding a lock. During fast scroll where `cachedHighlights()` is called per-cell, this will cause jank.

**Fix:** Use a doubly-linked list with a dictionary mapping key → node for O(1) LRU operations. Or use `NSCache` (which is already thread-safe and LRU-ish).
