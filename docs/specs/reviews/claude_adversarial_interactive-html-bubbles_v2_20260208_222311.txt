## Adversarial Review: Interactive HTML Bubbles Spec

### BLOCKERS (Must fix before implementation)

**None identified.** The spec is solid on the critical security properties.

---

### MAJOR (Fix before shipping to production)

#### 1. **Callback Ordering & Delivery Guarantees Unspecified** (Lines 77-107)

The spec says the client sends `interactive-callback` over the existing WebSocket, but doesn't specify:
- **At-least-once vs at-most-once vs exactly-once?** If the WS disconnects mid-callback, does the client retry? Does the provider deduplicate?
- **Ordering guarantees:** If a user taps Button A then Button B in quick succession, are callbacks guaranteed to arrive in order? WS is ordered, but if rate limiting drops callbacks (line 136: "excess silently dropped"), does that break ordering expectations?
- **Ack/nack:** No acknowledgment mechanism. If the provider or agent is slow, the client has no way to know the callback was received. For simple buttons this is fine, but for forms with important data, silent drops are a real problem.

**Concrete fix needed:** Add a short section specifying:
1. Callbacks are fire-and-forget (at-most-once) with rate limiting
2. Rate-limited callbacks are silently dropped (already stated) — make clear this means NO ordering preservation under rate limiting
3. If exactly-once is ever needed, that's v2 with explicit ack/retry

#### 2. **Rate Limit Drop Behavior Creates Silent Failures** (Line 136)

"Max 10 callbacks per second per bubble (excess silently dropped)" — this is reasonable, but:
- **JS has no way to know its callback was dropped.** A form submission could vanish with zero feedback.
- **Recommendation:** Instead of silent drop, consider queueing with a cap (e.g., queue up to 20, then drop). Or: expose a `window.clawline.callbackQueued` / `callbackDropped` event so JS can react.

**Minimum fix:** Document that agents should design interactions assuming callbacks can be lost. Or add a simple JS callback for drop notification.

#### 3. **`_resize` Allows Unbounded Layout Churn** (Lines 203-204)

The spec says `_resize` triggers "one additional measure-lock-invalidate cycle, then locks again." But:
- Can JS call `_resize` repeatedly? If so, every call triggers a layout invalidation
- There's no mention of a limit or rate on `_resize` calls

**Concrete fix:** Add: "_resize can be called at most once per bubble. Subsequent calls are ignored." Or rate limit to 1 per 5 seconds.

#### 4. **No Callback Idempotency Key** (Lines 77-99)

If the WS connection flaps and the client has reconnection logic, could a callback be sent twice? The callback message has `messageId` (referring to the original interactive message), but no unique callback ID to dedupe.

**Fix:** Add `callbackId: UUID` to the callback message for optional server-side deduplication if this becomes a problem. Can be v2 if you're confident WS reconnection logic won't cause double-sends.

### MINOR (Should fix, not blockers)

#### 5. **CSP Injection Location Unspecified** (Lines 146-151)

"The client injects a CSP meta tag wrapping all HTML before loading" — but where exactly?
- Before `<head>`? Inside `<head>`? What if the HTML has no `<head>`?
- Edge case: what if agent HTML already has a CSP meta tag with more permissive rules? First CSP wins, but behavior should be documented.

**Fix:** Specify: "CSP meta tag is injected as the first child of `<head>`, or prepended to `<html>` if no `<head>` exists. Client's CSP takes precedence over any agent-supplied CSP."

#### 6. **Memory Warning Teardown/Recreate Cycle** (Lines 242-243)

"On `didReceiveMemoryWarning`, tear down offscreen interactive bubble WKWebViews. They recreate on next scroll-into-view."
- This is fine, but: does the recreated bubble remember any JS state? (No, obviously — non-persistent store + fresh WKWebView)
- Should clarify: if user was mid-form in an offscreen bubble, memory warning destroys state. User will see fresh form on scroll back.

**Fix:** Add note: "State is not preserved across teardown — users may need to re-enter form data if memory pressure causes offscreen bubble teardown."

#### 7. **`_close` Summary Text Not Sanitized** (Lines 122-124)

The `_close` action has an optional `summary` string that replaces the bubble. This is controlled by the agent's HTML (which is trusted), but:
- Is there a length limit on summary?
- Can it contain arbitrary Unicode/emoji (presumably yes)?

**Fix:** Add: "Summary text max 500 characters. Excess is truncated."

#### 8. **Test Plan Missing** (Entire doc)

The spec is implementation-focused with no test plan. For the review request's completeness check, here's what I'd expect:

**Unit tests:**
- CSP injection with various malformed HTML inputs
- Size limit enforcement (provider + client)
- Callback validation (action length, data size, rate limit)
- `_resize` limit enforcement
- Error/crash recovery state machine

**Integration tests:**
- Full callback round-trip: JS → native → WS → provider → agent
- Rate limit behavior (send 20 callbacks in 1 second, verify 10 dropped)
- Memory warning teardown/recreate cycle
- Unknown version handling

**Security regression tests:**
- Verify no network requests escape (use Charles/mitmproxy)
- Verify navigation is fully blocked (test `window.open`, `location.href`, meta refresh, javascript: URLs)
- Verify process isolation from LinkPreviewView

**Performance tests:**
- 50 interactive bubbles in conversation — scroll performance, memory

### READY?

**Conditional GO** — ready for implementation with these changes:

1. **Required before implementation:**
   - Clarify callback delivery semantics (at-most-once, no ordering under rate limiting)
   - Cap `_resize` to one call per bubble lifetime

2. **Required before production:**
   - Add callback drop notification to JS (or document design-for-loss expectation)
   - Specify CSP injection location precisely
   - Add test plan to spec (or separate test spec)

3. **Defer to v2:**
   - Callback idempotency key
   - WKWebView pooling (if perf problems emerge)

The security model is sound. Defense-in-depth is well-designed. The sizing protocol learned from T020/T026/T028. The main gaps are around callback reliability semantics and the `_resize` loophole.
