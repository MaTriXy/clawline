# Adversarial Review: AI-Powered Salient Point Highlighting

## BLOCKERS

### 1. UTF-16 Offset Reliability is Fundamentally Broken

**The spec assumes the Foundation Models API will return accurate UTF-16 offsets into arbitrary text. This is almost certainly wrong.**

LLMs are notoriously bad at character/byte-level indexing tasks. Even with explicit instructions, models:
- Count characters inconsistently (especially with emoji, combining characters, multi-byte Unicode)
- Confuse grapheme clusters with code units
- Drift during tokenization (the model sees tokens, not UTF-16 offsets)

The `@Guide` annotations with `.range(0...100000)` don't help—they constrain the *value*, not the *accuracy*.

**Impact**: Highlights will land on wrong text spans constantly, creating a broken user experience.

**Fix required**: Don't ask the model for offsets. Instead:
- Have the model return the *exact substring text* it wants highlighted
- Use `NSString.range(of:)` or similar to find those substrings in Rendered Text
- Handle ambiguity (multiple occurrences) with heuristics or by asking for surrounding context

### 2. No "Rendered Text" Definition for Mixed Content

The spec says "Rendered Text" is the exact string shown in the bubble, but `MessagePresentation` can have multiple parts (`.text`, `.markdown`, `.code`, `.table`). 

**Unspecified**:
- Is Rendered Text the concatenation of all parts? Just text/markdown parts?
- What separator between parts (if any)?
- Are code blocks included in analysis but excluded from highlighting?

If you concatenate parts with, say, newlines, but the model returns spans that cross part boundaries, those spans are invalid.

**Fix required**: Define precisely:
- Which parts contribute to Rendered Text
- The exact concatenation logic (or that each part is analyzed separately)
- How spans are mapped back to individual parts

### 3. Cache Key Race Condition

The cache key includes `renderedTextHash`, but `renderedTextHash` is computed from Rendered Text, which is computed *during rendering*. 

**Problem**: If the rendering pipeline changes (font substitution, URL stripping logic update, markdown parser fix), the same `Message.content` produces different Rendered Text and different hashes.

But the spec doesn't say when/where the hash is computed. If it's computed at generation time and again at application time, a mismatch causes cache misses or—worse—applying spans to different text than was analyzed.

**Fix required**: Either:
- Store Rendered Text alongside spans in the cache and verify it matches before applying
- Or compute hash once and store it with the message, invalidating only on explicit model changes

### 4. Foundation Models API Availability Unknown

The spec assumes `SystemLanguageModel.default.availability` exists and works as described. But:
- Is this the actual API? The Foundation Models framework is new (iOS 26)
- What happens on devices without Apple Intelligence hardware?
- What if the user hasn't enabled Apple Intelligence in Settings?

**The spec gives no fallback behavior** beyond "render normally"—but how do you detect unavailability at runtime vs. a transient error?

**Fix required**: 
- Verify the actual API surface exists
- Define detection logic: one-time check at startup? Per-generation check? Cached capability flag?
- Distinguish "not available ever" from "temporarily unavailable" (e.g., thermal throttling)

### 5. No Mention of Session Lifecycle

`LanguageModelSession` presumably has a lifecycle. The spec doesn't address:
- When to create/destroy sessions
- Whether sessions are reusable across multiple generations
- Memory/resource implications of holding a session open
- Thread safety of session objects

**Fix required**: Define session management strategy explicitly.

## SERIOUS ISSUES (High Risk, Potentially Blocking)

### 6. Span Overlap Resolution is Underspecified

The spec says "drop invalid or overlapping spans deterministically" but doesn't define the algorithm.

- If span A (0-10, bold) overlaps span B (5-15, italic), which wins?
- Is "deterministic" based on array order? Confidence score? Kind priority?
- What if the model returns adjacent spans that should be merged?

Without a defined algorithm, implementations will diverge and behavior will be confusing.

### 7. "Do Not Exceed ~30% of Text" is Unenforceable

You can tell the model this, but you cannot make it comply. What's the fallback?

- Silently drop excess spans? (Lossy, which ones?)
- Apply all of them anyway? (Defeats the purpose)
- Reject the entire result? (Wasteful)

### 8. Inline Code Detection is Fragile

"Skip spans that intersect inline code runs (monospace / `.traitMonoSpace`)"

But:
- `ChatMarkdownRenderer` may not consistently apply `.traitMonoSpace`
- Inline code detection via font traits is heuristic, not semantic
- What about `NSAttributedString` produced by other renderers or legacy code paths?

**Risk**: Highlights applied inside code spans, breaking visual parsing.

### 9. Rate Limiting Strategy is Vague

"Max 1-2 in-flight generations" and "debounce batches" are mentioned but not specified:
- What's the exact debounce interval?
- How is the queue prioritized? (Visible messages first? Chronological?)
- What happens if the user scrolls through 500 messages rapidly?

Without specifics, implementations will either starve (never catching up) or spike (ignoring limits).

### 10. No Error Handling Beyond "Render Normally"

What happens when:
- Generation times out?
- The model returns malformed JSON?
- The model returns empty spans for every message?
- Disk cache is corrupted?

"Render normally" hides these failures, making debugging impossible.

## MODERATE ISSUES

### 11. Algorithm Version Is Global

If you bump `algorithmVersion`, *all* cached highlights are invalidated. For a user with thousands of messages, this could cause:
- Massive regeneration storms on next app launch
- Significant CPU/battery drain
- Potential rate limiting from Apple Intelligence APIs (if any)

Consider: per-message lazy invalidation, or a migration path that preserves "good enough" old results.

### 12. Message Immutability Assumption May Be Wrong

The spec says "assume messages are immutable once sent" but:
- What about edited messages (if the app supports editing)?
- What about messages that get "cleaned up" (whitespace normalization, link extraction)?

If messages can change, the `renderedTextHash` mismatch will cause silent failures.

### 13. Truncation Strategy for Long Messages is Undefined

"Cap Rendered Text to a maximum character/UTF-16 length" and "prefer 'top of message' + 'last lines' sampling" are mentioned but:
- What's the actual length limit?
- How are top/bottom portions joined? (Newline? Ellipsis marker?)
- How do you handle spans that reference the middle (now missing)?

### 14. Memory LRU Size is Arbitrary

"300-800 entries" with no justification. This could be:
- Way too small for users with long chat histories who scroll frequently
- Way too large for memory-constrained devices

Should be tied to memory pressure signals or configurable.

### 15. Disk Cache Strategy Has File Count Risk

"File-per-message" in app caches for users with thousands of messages means thousands of small files. Some file systems don't handle this well. The "SQLite if file count becomes problematic" fallback should be the default.

### 16. No Accessibility Considerations

Bold/italic are visual-only. VoiceOver users won't benefit. Should spans also update `accessibilityLabel` to indicate emphasis (e.g., "important: " prefix)?

### 17. No Privacy Review

The spec processes user message content through Apple Intelligence. Even though it's on-device:
- Are there Apple telemetry implications?
- Does this comply with the app's privacy policy?
- What happens if the user has Analytics disabled?

## NICE-TO-HAVES / MINOR ISSUES

### 18. Confidence Score is Unused

`SalientSpan.confidence` is defined but the spec never says what to do with it. Either use it (e.g., only apply spans above threshold) or remove it.

### 19. Kind Enum is Unused

`SalientSpan.kind` is optional and never referenced in the rendering logic. If it's just for debugging, say so explicitly.

### 20. Telemetry is "Optional"

For a feature with this many failure modes, telemetry should be required, not optional. At minimum: cache hit rate, generation failure rate, span validity rate.

### 21. No Versioning Story for Cache Format

If you change `SalientHighlights` or `SalientSpan` codable structure, old cached files become unreadable. Need either a schema version or robust decoding that handles missing/extra fields.

### 22. No Mention of Localization

Does this work for non-English messages? Arabic (RTL)? Chinese (no spaces)? The prompt/instructions assume English-centric "phrases."

### 23. Testing Plan Lacks End-to-End

Unit tests are good, but there's no mention of:
- Manual QA checklist
- UI snapshot tests for highlighted bubbles
- Performance benchmarks (generation latency, scroll frame rate)
