Clawline Terminal Bubble Tool API Architecture

Formal source of truth for what terminal bubbles are for, how callers request them, and what Clawline must preserve when routing terminal sessions.

StatusApproved; Tightening Applied
FeatureT001 Terminal Bubbles
Owner SurfaceClawline
Last Updated2026-05-17

Purpose

Terminal bubbles exist so an agent or tool caller can put a live terminal session inside the current Clawline conversation while naming the exact machine and terminal session identity that should back it.

This is not a decorative attachment type and not a generic remote shell picker. It is a product contract: a caller requests a terminal bubble for a specific destination and session identity, the provider provisions or attaches exactly that target, and the user sees a live terminal surface in the message stream.

The central invariant is simple: the bubble must represent the terminal the caller asked for. If Clawline silently creates a different tmux session, routes through a provider-global fallback, or forces callers to construct raw descriptors to get the right identity, the feature has failed.

Product Invariants

IDInvariantMeaning
TB-01Caller intent is authoritative.The structured tool request names the destination and, when supplied, the terminal/tmux session identity. Provider defaults may fill omissions, but may not override explicit caller intent.
TB-02Destination is per bubble.A terminal bubble request carries its own destination. A provider-global tmux host setting is not allowed to be the hidden routing authority for new bubbles.
TB-03Session identity is first-class.Callers must be able to name the terminal session identity through the structured API. Raw descriptor construction is an escape hatch, not the intended product path.
TB-04Unnamed requests stay in the login shell.If the caller supplies no terminal session name, Clawline must connect to the requested destination and leave the user in the connected login shell. It must not automatically create or attach to a generated tmux session.
TB-05No false success.If the requested destination or session cannot be reached/provisioned, the bubble must fail visibly for that exact request. It must not fall back to another machine or a new unrelated session.
TB-06The provider owns terminal transport.The client renders and interacts with the terminal bubble. The provider owns SSH/tmux attach/provisioning and proxies PTY bytes over the terminal WebSocket.
TB-07Descriptor truth survives reconnect.The message attachment descriptor and provider session record must preserve destination and session identity so reload/reconnect/auth attach to the same target.

Correct Tool API

Terminal bubbles are sent through the Clawline message tool as a structured sendAttachment action. The API must support terminal-specific fields directly; callers must not need to hand-build the terminal descriptor JSON to express the normal product path.

Required API Shape

{
  "action": "sendAttachment",
  "contentType": "application/vnd.clawline.terminal-session+json",
  "destination": {
    "address": "clu@racter"
  },
  "terminalSession": {
    "name": "clawline-font-check"
  },
  "title": "racter · clawline-font-check"
}
IDFieldRequirement
API-01actionMust be sendAttachment for terminal bubble creation.
API-02contentTypeMust be application/vnd.clawline.terminal-session+json. This MIME type is how clients identify terminal bubbles.
API-03destination.addressRequired for new terminal bubbles. It names the concrete provider-reachable target, such as clu@racter, mike@eezo, or another authorized SSH/tmux endpoint.
API-04terminalSession.nameOptional. When supplied, it is the caller-requested terminal/tmux session identity and must be preserved. When absent, no tmux session is implied; the terminal opens at the destination login shell.
API-05titleOptional presentation text. It may default from destination/session, but it must never be used as routing truth.

Compatibility Alias

terminalSession.name is the documented product-facing field. It names the backing terminal/tmux session the caller wants the provider to attach or provision.

If the implementation accepts lower-level legacy names, terminalSessionId and tmuxSessionName are compatibility aliases only. Alias resolution must be strict: if terminalSession.name, terminalSessionId, and/or tmuxSessionName are supplied together and disagree, the request must fail visibly before emitting a bubble. The provider must not silently choose one value.

IDAlias RuleAcceptance
API-06terminalSession.name has precedence only when aliases are absent or equal.Equal duplicate values normalize to one requested session name.
API-07Conflicting alias values fail the request visibly.A request with terminalSession.name: "a" and tmuxSessionName: "b" returns a structured error and creates no misleading terminal bubble.
API-08tmuxSessionName remains an accepted compatibility alias, not the documented product field.New T001 callers use terminalSession.name; old callers are tolerated only when unambiguous.

Descriptor Contract

The provider emits a terminal-session document attachment. The descriptor is the durable artifact the client renders and uses to authenticate the separate terminal WebSocket.

{
  "version": 3,
  "terminalSessionId": "termmsg_9a0f6b",
  "title": "racter · clawline-font-check",
  "destination": {
    "address": "clu@racter"
  },
  "terminalSession": {
    "name": "clawline-font-check",
    "provisioning": "attach_or_create"
  },
  "provider": {
    "baseUrl": "https://provider.example",
    "wsPath": "/ws/terminal"
  },
  "capabilities": {
    "interactive": true,
    "supportsBinaryFrames": true,
    "supportsResize": true,
    "supportsDetach": true
  },
  "auth": {
    "mode": "chat_token"
  }
}
IDDescriptor RuleReason
DESC-01version: 3 means destination-and-session-aware terminal descriptor.Version 2 only established explicit destination; version 3 closes the structured session identity gap.
DESC-02terminalSessionId remains the protocol/auth lookup id.The terminal WebSocket can stay keyed by an opaque message/session id without conflating that id with the tmux name.
DESC-03terminalSession.name is the requested backing session identity.This is what the provider attaches/provisions on the destination. It may equal the auth id, but does not have to.
DESC-04The descriptor must preserve destination.address and terminalSession.name.Reload, reconnect, and historical rendering need the same target truth the original request carried.
DESC-05The client must not infer destination or session identity from title, provider URL, or auth id.Those fields are not routing truth.
DESC-06Legacy v1/v2 descriptors are visibly legacy or unknown when target truth is missing.Fallback behavior may preserve old bubbles, but new structured T001 requests must not use legacy/unknown descriptors as product evidence.

Provider Behavior

IDRequirementAcceptance
PROV-01On structured request, validate destination.address and optional terminalSession.name.Invalid, unauthorized, or internally conflicting target fields fail before a misleading bubble is emitted. The structured error includes the supplied destination.address and, when supplied, terminalSession.name.
PROV-02Persist a session record containing owner, chat session, descriptor id, destination address, requested terminal session name, title, timestamps, and lifecycle state.Provider restart or terminal reconnect can restore the exact target.
PROV-03Attach/provision against the requested destination and terminal session name.A request for clu@racter + clawline-font-check uses attach_or_create by default: attach to that named session if it exists, otherwise create that named session on the requested destination. If either path fails, fail visibly for that exact target.
PROV-04Do not create a tmux session when terminalSession.name is absent.Unnamed requests connect to the destination login shell. Random/generated tmux session names are not a fallback.
PROV-05Do not use provider-global fallback to satisfy explicit destination/session requests.Fallback can support legacy v1/v2 descriptors only when the descriptor lacks required target truth and the UI labels that limitation honestly.
PROV-06Return terminal errors with the requested destination/session in structured diagnostic context.Every structured failure includes destination.address and, when supplied, terminalSession.name. Flynn and agents can tell whether the exact requested target failed instead of guessing from title/text.
PROV-07Reserve stricter provisioning modes for future explicit API expansion.attach_existing and create are not v3 defaults. They may be added later only if callers need stricter semantics than attach_or_create.

Client Behavior

IDRequirementAcceptance
CLIENT-01Detect terminal bubbles by MIME type.application/vnd.clawline.terminal-session+json renders as a terminal bubble, not a generic document preview.
CLIENT-02Use the terminal WebSocket for PTY traffic.Chat WebSocket and terminal WebSocket remain separate; binary PTY traffic is not routed through chat messages.
CLIENT-03Authenticate terminal WebSocket with descriptor protocol id and normal chat auth.The provider resolves the persisted destination/session record. The client does not resend a second routing hint at auth time.
CLIENT-04Display target truth from descriptor fields only.Destination/session labels match the actual descriptor or are explicitly unknown for legacy bubbles.
CLIENT-05Resize and input are sent only after terminal auth is ready.No race where resize/input causes a disconnected terminal or false reconnect loop.

Non-Goals

  • Not a Surf Ace native-pane launch API. Surf Ace can review whether this API is sane, but T001 is Clawline terminal bubbles.
  • Not a registry or picker for all possible machines. The initial contract is direct destination strings authorized by provider policy.
  • Not a client-side SSH feature. Clawline clients render; providers connect.
  • Not proof if it works only through raw descriptor injection or temporary harness state.

Verification Matrix

V-TB-04Reload/reconnect.Existing bubble reconnects to the same destination/session from persisted descriptor/record truth.
V-TB-05Real client path.Evidence comes from normal message.sendAttachment structured API through a real Clawline client/provider path, not demo insertion or raw DB state.
V-TB-06Conflicting alias fields.Provider returns a visible structured error containing the requested destination/session fields and emits no terminal bubble.
