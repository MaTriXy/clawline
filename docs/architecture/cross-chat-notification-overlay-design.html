Cross-Chat Notification Overlay Design

Clawline architecture/design source of truth for T307 notification bubbles, dock behavior, buttons, and hotkeys. Created 2026-05-16.

Decision: cross-chat notifications are a volatile overlay for assistant output from other chat sessions. They never write into the current transcript, never replace visible controls with shortcut-only behavior, and always route replies/status to the associated source chat.

Scope

This document covers the notification overlay side of T307: bubble creation, stacking, docking, visible actions, reply mode, keyboard shortcuts, and navigation behavior. Prompt-field @ mention routing remains defined by the T307 cross-chat spec, but the same routing invariant applies here: sends belong to the target chat, not the currently viewed chat.

Product Invariants

  • Notifications are assistant-message-only. User, system, and tool messages do not create bubbles.
  • Notifications are volatile and in-memory. They do not persist across app relaunch, process restart, or session reset.
  • Each source chat has at most one notification bubble. Repeated assistant messages from that source append into the same bubble.
  • Notification replies are real sends to the notification's associated chat. They appear only in that chat and must drive that chat's normal pending/running/success/failure status path.
  • The current chat must not receive a local user row, synthetic status row, transcript echo, or dot/status color change for a cross-chat reply it did not receive.
  • Visible buttons are required. Hotkeys are accelerators, not substitutes for on-screen Dismiss and Reply controls.

Overlay Layering

  • The notification stack renders in its own overlay layer above the current chat transcript and composer.
  • The chat session name toast renders above notification bubbles.
  • Bubbles anchor near the upper-right safe visible area, leaving only normal safe-area/chrome margin. They should not start noticeably down the screen.
  • On compact screens, the stack must cap by actual available height. Ten visible bubbles is only the large-screen upper bound, not a hard count.
  • Layout must avoid incoherent overlap with the composer, transcript controls, or app chrome.

Bubble Content

  • A bubble represents one source chat and shows the source chat label, optional shortcut slot, latest assistant content, older accumulated content, and visible action buttons.
  • Newest content appears above older content inside the bubble. Older content is pushed downward and may move below the visible fold.
  • Streaming updates for the same assistant message update that message entry rather than creating repeated entries.
  • Short bubbles shrink vertically to fit content without excess bottom padding.
  • Long content scrolls inside the bubble's max-height area; transcript scroll must not be stolen.
  • The max height is tuned for device review: latest reduced cap was too short, so use about one third more than that reduced cap while still keeping long content internally scrollable.
  • Notification header chrome is compact, with reduced vertical padding above and below header content.

Stack And Overflow

  • The visible stack is ordered by notification recency, with updated source bubbles re-entering the ordering when new assistant output arrives.
  • When more bubbles exist than fit, the oldest visible bubble animates off-stack into hidden overflow. It is not dismissed.
  • Dismissed visible bubbles reveal hidden overflow bubbles according to the current stack ordering.
  • Navigating into a source chat dismisses only notifications associated with that chat. Unrelated notifications remain.
  • Cmd-minus clears visible and hidden notification backlog.

Dock And Peek Mode

  • The notification stack supports a right-edge docked/peek mode.
  • Swiping notifications right collapses the whole stack off-screen to the right, leaving only left-edge peeks visible.
  • Tapping a peek or swiping left restores the full stack.
  • Cmd-\ toggles between normal visible placement and right-edge docked/peek mode using the same state transition as swipe collapse/restore.
  • New bubbles animate in from off-screen right. Leaving, dismissing, hiding, and docking animate toward off-screen right.
  • Docking is a display state, not dismissal. Hidden/docked bubbles retain their associated source chat and content until dismissed, cleared, navigated into, or reset.

Visible Controls

Dismiss

Icon-only control using SF Symbols/SF fonts on Apple platforms and the platform symbol system on Electron/web. Tapping removes that notification. Holding or long-pressing opens a standard confirmation dialog offering to clear all notifications.

Reply

Icon-only toggle that opens an inline reply field inside the bubble. Tapping Reply again while active cancels reply mode. Escape also cancels without sending.

Body Tap

Tapping notification content outside controls navigates to the notification's source chat and dismisses notifications for that source only.

Reply Mode

  • Reply mode resizes the bubble to the reply interface instead of preserving the prior message-content height.
  • The inline reply field has a visible send button with a tappable hit target. Software keyboard Return on iOS submits the reply.
  • Accepted reply sends dismiss that notification after the send is accepted by the normal send path.
  • Reply sends appear only in the associated chat and update only that chat's status/dot color through the same source of truth as normal sends.
  • Failures use the existing send failure path for the reply UI without creating a current-chat transcript row.

Shortcut Slot Assignment

  • Only visible notifications receive numeric shortcut slots.
  • Slots are reassigned as notifications appear, dismiss, hide, reveal, dock, or restore.
  • Shortcut numbers are shown only when hardware-keyboard shortcuts are available. On iOS/iPadOS, hide numbers unless a hardware keyboard is detected. On macOS/Catalyst, assume shortcuts are available unless platform state says otherwise.
  • If no bubble is explicitly active, Cmd-J and Cmd-K target the top visible notification. If that bubble has no scrollable overflow, do nothing and do not scroll the transcript.

Platform Presentation

  • Apple platforms should follow standard Liquid Glass presentation rules for notification chrome, controls, dialogs, and hit targets.
  • Controls remain visually clean and icon-only. Do not add visible background circles solely to enlarge hit targets.
  • Electron/web should use the nearest platform-native symbol system and preserve the same behavior model.
  • Animations should be spatially consistent: notifications enter from the right and leave/collapse toward the right.

Acceptance Checklist

  • Assistant output from another chat creates an upper-right bubble while viewing the current chat.
  • Repeated assistant output from the same source stays in one bubble with newest content above older content.
  • Short content shrinks; long content scrolls internally; compact screens show only what actually fits.
  • Dismiss and Reply are visible, icon-only, tappable controls.
  • Reply can be sent with the button or iOS software keyboard Return, can be canceled with Escape or active Reply toggle, and dismisses the notification after send acceptance.
  • Reply sends and cross-chat sends update the associated chat's normal status path and do not mutate the current chat's transcript/status.
  • Body tap navigates to source chat; source-chat navigation clears only that source's notifications.
  • Overflow hides without dismissal and reveals after visible dismissal.
  • Right-swipe and Cmd-\ dock the stack into right-edge peeks; tap/switch restores it.
  • Number shortcuts navigate/reply/dismiss according to the matrix, and numbers display only when hardware-keyboard shortcuts are available.
  • Cmd-J/Cmd-K scroll notification content without moving the transcript or changing notification order.
  • The chat session name toast renders above notification bubbles.
